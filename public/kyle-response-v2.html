Ask 3

SOC 2 / SSO / Audit Logging

Enterprise procurement gates. Non-negotiable for any serious buyer.
Honest 180-Day Timeline — Revised

The original response listed SOC 2 readiness at Day 76-90. That was wrong. Here is what SOC 2 actually requires:

Day 1–90
Readiness

Controls implemented. Policies documented. Auditor engaged. This is what "SOC 2 ready" actually means — not certified, but fully prepared for observation to begin.

Day 91–180
Type I Certification

Observation period begins. Evidence collected. Type I certification is a point-in-time audit, but auditors require a documented period of operating controls before they will sign off. Cannot be compressed.

Day 181+
Type II (if required)

Type II requires 6-12 months of operating history. This is a post-deployment milestone, not a pre-deployment gate.

Day-by-day build sequence:

  • Day 1-30: Immutable audit logging for all agent actions and API calls.
  • Day 31-60: Role-based access control (RBAC) — Admin, operator, LO, underwriter roles. SSO via OIDC/SAML (Auth0 or Okta).
  • Day 61-90: Data handling policies finalized. PII classification documented. Encryption at rest and in transit verified. Auditor engaged.
  • Day 91-180: Observation period + evidence collection. Type I certification target at Day 180.
For Rapid-internal deployment: external SOC 2 is not required in Phase 1. Rapid is the customer and owner. Borrower data never leaves Rapid's environment. Audit logging and RBAC (Days 1-60) are the real gate for internal pilot use.

Section 4 — Query Routing Logic
Routing

How Queries Are Routed — And Why It Matters

Designed In

Not all guideline questions are equal. Routing the wrong query to the wrong model either wastes money (Opus for a simple lookup) or produces bad output (Haiku for a nuanced cross-agency interpretation). Here is the actual routing logic:

Query TypeModelLatencyInput CostExample
Simple factual Haiku <1s $0.0008/1K What is the max DTI for FHA?
Complex interpretation Sonnet 1–2s $0.003/1K How does FHA treat student loans in IBR for qualifying income?
Cross-agency / novel Opus 2–3s $0.015/1K Which agency is best positioned for this borrower profile and why?

Safety net: Low-confidence Haiku answers automatically escalate to Sonnet before reaching the LO. The LO never sees a raw AI answer — every response includes source citations for human verification.

Wrong routing risk: If Haiku routes a complex question and returns low confidence, it escalates. If it returns high confidence on a wrong answer, the source citation requirement catches it — the LO is directed to the exact guideline section and can verify. AI is a lookup accelerator, not an authority.
Section 5 — Cache Invalidation Strategy
Cache

When Guidelines Change, What Happens

Version-Tagged and Monitored

Stale guideline data in a mortgage context is not a minor issue. This is the invalidation architecture:

  • Version tagging: Every cached guideline chunk is tagged with the effective date of the source document. FHA guidelines are versioned; FNMA selling guide is edition-tracked.
  • Daily automated check: System pings source PDF URLs daily and compares checksums against cached versions. Any change triggers immediate cache invalidation and re-embedding.
  • LO-visible indicator: Every answer displays "Based on guidelines effective as of [date]." If guidelines were updated within the last 7 days, a yellow indicator appears on the answer.
  • Cache TTL: 24 hours as a floor. Immediate invalidation on update detection.
  • Manual override: Operator can force a full re-index on any agency at any time. Takes approximately 4-6 minutes per agency.
Honest gap: automated update detection relies on source PDFs being accessible at stable URLs. FHA and FNMA maintain consistent URL patterns; USDA and VA are less predictable. For USDA and VA, the daily check falls back to a human review flag if the URL is unreachable.

Section 7 — Security and Encryption
Security

Full Security Architecture

Documented Standard

Data in transit: TLS 1.3 on all endpoints. Already in place via Cloudflare. No plaintext traffic.

Data at rest: AES-256 encryption for any stored data. Implementation required before first external deployment.

Borrower PII handling — critical:

  • Borrower PII is never sent to the Claude API. The AI only receives guideline text (public documents from FHA, FNMA, USDA, VA, FHLMC).
  • Income calculations run locally in a deterministic rules engine. No income figures, borrower names, SSNs, or loan amounts leave the local environment.
  • The LO inputs a guideline question. The AI returns a guideline answer with a source citation. No borrower data in the loop.

Credential management: Current state uses .env files (acceptable for single-operator development). Pre-deployment migration to AWS Secrets Manager or HashiCorp Vault. Rotating credentials on a defined schedule.

Incident response plan:

  • Incident log maintained and available for client review
Section 8 — Anthropic API Outage Failover
Failover

What Happens If Claude Goes Down

Designed for Resilience

The income calculation engine does not use Claude. It is a deterministic rules engine. DTI calculations, qualifying income calculations, asset seasoning checks — all of these work without any AI API. An Anthropic outage does not affect income calculations at all.

Guideline query resilience:

  • Frequently-asked guidelines are cached locally after first query. Common FHA questions return from cache without an API call.
  • If Claude is unreachable: cached answers are served immediately. Complex or uncached queries are queued and LOs are notified with an estimated resolution time.
  • Anthropic guarantees 99.9% uptime on their API. Historical outages are rare and brief.

Model portability: The RAG architecture is model-agnostic. The vector store is independent of the inference provider. If a permanent switch is needed, OpenAI or Google Gemini can be hot-swapped with a single configuration change. The guideline documents do not move. The LO interface does not change. Only the inference call changes.

Practical impact of a 1-hour Claude outage: LOs get cached answers for common questions. Unusual questions queue. Income calculations are unaffected. This is a manageable degraded state, not a full service outage.

Section 12 — Human Oversight in Daily Operations
Human Oversight

AI Does Not Make Underwriting Decisions

Designed In from Day One

This is not a box to check for regulators. It is a design principle.

  • Source citations required: Every AI-generated guideline answer includes a source citation (FHA handbook section, FNMA selling guide chapter, etc.). The LO verifies the answer against the source. The AI is a lookup accelerator, not an authority.
  • Income calculations are deterministic: The rules engine is auditable, reproducible, and produces the same output for the same input every time. An underwriter can trace any income calculation step by step. No AI black box in the math.
  • Confidence flagging: Low-confidence answers are flagged for human underwriter review before the LO acts on them. The system knows what it does not know.
  • Operations log: Daily log of all queries, answers, confidence scores, and flags. Reviewed by operator. Provides an audit trail for any deal-level review.
  • Monthly architecture review with Kyle (proposed): External technical oversight by someone who has built mortgage technology at this level.
